# LP-0013: Token program improvements: authorities + associated token accounts + wallet tooling

**`Logos Circle: <N/A>`**

## Overview
This prize targets practical, production-oriented improvements to the LEZ Token program so it can support real-world Token usage. The current example is intentionally minimal; this challenge focuses on adding a small set of high-impact capabilities that wallets and apps expect: controlled minting via authorities, deterministic “Associated Token Accounts” (ATAs), and basic wallet/CLI commands to manage them.

[Solana’s SPL Token ecosystem](https://solana.com/docs/tokens/basics/create-token-account) is a useful reference point here: it separates *mints* from *token accounts* and commonly uses ATAs as a canonical, deterministic token account per wallet+mint, which makes balances easy to find and improves UX.

## Motivation
Without variable supply (minting authorities) and deterministic token accounts, the ecosystem quickly runs into friction:
- token issuers can’t support common supply models (fixed supply, capped supply, permissioned minting, revocable minting),
- wallets can’t reliably discover where a user holds a given token without heuristics,
- user UX suffers (manual account creation, confusing account selection, brittle integrations).

A competitive prize is the right mechanism because there are multiple valid designs (authority model, ATA derivation scheme, compatibility with future “token extensions”), and the ecosystem benefits from evaluating implementations side-by-side for minimalism, correctness, and developer experience.

## Success Criteria
- [ ] Authority-based minting (variable supply): Implement a mint authority model for LEZ tokens that supports:
  - mint authority set at token initialisation,
  - minting by the authority,
  - authority rotation and/or revocation (e.g., set to `None` to make supply fixed).
- [ ] Associated Token Account (ATA) mechanism: Implement an ATA program/pattern such that:
  - there is a deterministic token account address for a given `(owner, mint)` pair,
  - the derivation is documented and stable,
  - creation of an ATA is idempotent (safe to call repeatedly),
  - token accounts created via the ATA mechanism are clearly identifiable as “associated token accounts”.
- [ ] Wallet/CLI command support: Provide user-facing commands (or a minimal wallet-facing library + CLI) for:
  - creating an ATA for a given mint,
  - querying the derived ATA address,
  - listing a wallet’s ATAs for known mints (at minimum via explicit mint list input),
  - transferring tokens using ATAs as defaults when available.
- [ ] Backwards-compatible baseline: Existing simple transfers (one sender/one recipient), mint/burn, and NFTs continue to work (or have a clear migration story and compatibility layer).
- [ ] Documentation and examples: Provide:
  - at least *two* example integrations (e.g., “fixed supply token with revoked authority” and “variable supply token with mint authority”),
  - a short “Wallet integration guide” describing how to derive/create/use ATAs.


## Scope


### In Scope
- Token program improvements:
  - mint authority (and optionally freeze authority if proposed, but not required),
  - ATA programme/pattern for deterministic token accounts,
  - minimal wallet/CLI tooling for ATA lifecycle.
- Clear specifications for:
  - authority semantics,
  - ATA derivation and ownership rules,
  - expected UX defaults (e.g., “use ATA if present; otherwise prompt/create”).
- Minimal, auditable implementation with strong tests.

### Out of Scope
- A full-featured token standard with dozens of optional extensions (though proposals may reference such ideas).
- A complete production wallet UI or hosted explorer/indexer.
- Complex compliance/allowlist systems, advanced fee mechanics, or on-chain DEX integration (those can be separate prizes).
