# LP-0006: Bitcoin–LEZ Atomic Swap Using Adaptor Signatures

**`Logos Circle: N/A`**

## Overview

The Logos ecosystem already has a working ETH–LEZ atomic swap using Hash Time-Locked Contracts ([eth-lez-atomic-swaps](https://github.com/logos-blockchain/eth-lez-atomic-swaps)). Bitcoin is a different challenge: while Bitcoin supports HTLCs via `OP_CHECKLOCKTIMEVERIFY`, adaptor signatures are a strictly better approach — they require no script beyond a standard Taproot key-path spend, leave no hash preimage on-chain, and make the swap transaction indistinguishable from any other Bitcoin payment.

This prize funds a trustless Bitcoin–LEZ atomic swap using Schnorr adaptor signatures: a Bitcoin-side protocol built on Taproot/BIP-340, and a new LEZ program that locks funds contingent on revealing the adaptor secret (a discrete logarithm), not a hash preimage.

## Motivation

Bitcoin is the most liquid and widely held digital asset. Enabling trustless BTC↔LEZ swaps without custodians, bridges, or wrapped tokens is a significant capability for the Logos ecosystem and a prerequisite for meaningful DeFi liquidity.

The HTLC approach (used in the existing ETH–LEZ swap) works on Bitcoin but has drawbacks: it requires a non-standard script, the revealed preimage links the two legs of the swap on-chain, and it is identifiable as an atomic swap by chain observers. Adaptor signatures solve all three: the Bitcoin transaction is a plain Taproot spend, no preimage appears on-chain, and the two legs are cryptographically linked without on-chain correlation.

The LEZ side requires a new program — unlike the ETH case, where a hash lock suffices, here the LEZ escrow must enforce a condition based on a discrete logarithm commitment (an EC point), not a hash. This is a new cryptographic primitive for LEZ.

A competitive prize is the right mechanism because the adaptor signature protocol, the LEZ-side escrow design, and the cross-chain coordination protocol all admit multiple valid approaches with meaningful trade-offs in complexity, safety, and UX.

## Protocol Sketch

The swap proceeds as follows (Alice holds BTC, Bob holds LEZ):

1. **Setup**: Alice generates a secret scalar `t` and publishes the adaptor point `T = t·G`.
2. **Alice pre-signs**: Alice creates a pre-signed Bitcoin transaction paying Bob, encrypted with `t` — a valid adaptor signature that Bob can complete only by learning `t`.
3. **Bob locks LEZ**: Bob locks LEZ tokens in a LEZ escrow program, redeemable by Alice upon revealing `t` (verified on-chain via a discrete log proof or equivalent), or refundable to Bob after a timelock (enforced client-side — see note below).
4. **Alice claims LEZ**: Alice submits `t` to the LEZ escrow, receives LEZ tokens. `t` is now observable on LEZ.
5. **Bob claims BTC**: Bob extracts `t` from Alice's LEZ claim, completes the adaptor signature, and broadcasts the Bitcoin transaction.

## Success Criteria

- [ ] A trustless BTC↔LEZ swap can be completed without any on-chain script beyond a standard Taproot key-path spend on the Bitcoin side.
- [ ] The Bitcoin swap transaction is indistinguishable from a normal Taproot payment to a chain observer.
- [ ] The LEZ escrow program correctly enforces release of funds upon proof of knowledge of the adaptor secret `t`, and supports refund after timelock expiry. Note: LEZ does not yet have on-chain block time, so the LEZ-side timelock is enforced client-side (the same approach used in [eth-lez-atomic-swaps](https://github.com/logos-blockchain/eth-lez-atomic-swaps)).
- [ ] The two legs of the swap are atomic: either both complete or both refund. There is no state where one party receives funds and the other does not.
- [ ] A reference integration is delivered: a working demo of a complete BTC↔LEZ swap on Bitcoin testnet and LEZ testnet.
- [ ] At least 5 complete BTC↔LEZ swaps are executed on Bitcoin testnet and LEZ testnet, involving at least 3 distinct counterparty pairs outside the submitting team.
- [ ] Full documentation and a clean public repository are delivered.

## Scope

### In Scope

- A Bitcoin-side adaptor signature implementation using Schnorr/BIP-340 and Taproot key-path spends, including pre-signing and signature completion logic.
- A new LEZ escrow program (Rust, Risc0) that locks funds contingent on revealing the adaptor secret `t`, with timelock-based refund.
- A cross-chain coordination CLI or script that guides both parties through the swap protocol.
- A reference integration on Bitcoin testnet and LEZ testnet demonstrating a complete swap.
- Documentation covering: adaptor signature construction, LEZ escrow design, the cross-chain atomicity argument, security assumptions, and known limitations.

### Out of Scope

- Lightning Network integration.
- Multi-hop or routed swaps.
- The ETH–LEZ HTLC pattern — this prize is specifically for the adaptor signature approach on Bitcoin.
- A polished consumer UI — a working demo is sufficient.
- Ongoing maintenance or security audits beyond initial delivery.
